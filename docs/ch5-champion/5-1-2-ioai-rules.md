## 先讲清楚：为什么"规则与工具"值得专门花一页

你可能觉得，比赛拼的是模型本事，规则不就是看几眼的事吗？恰恰相反。IOAI 是一场**限时、限算力、限环境**的现场赛——很多选手不是输在不会做，而是输在"会做却交不上去"：本地跑得好好的代码，到了赛场环境里因为版本对不上直接报错；或者显存爆了，模型根本加载不进去；又或者临到结束才发现提交格式写错，白白丢了分。所以在你冲刺真题之前，我们先把这些"场外因素"一次性扫清，让你上场时心里有底。

下面的具体数字（GPU 型号、显存、库版本）每年官方会在赛前手册里微调，**正式数据一律以当年的 Competition Manual 为准**。这里给的是近几届的稳定面貌，帮你建立预期、定好备战策略。

## 允许的库：PyTorch 是主角，但别想着联网装包

IOAI 的技术栈是围绕 **PyTorch** 搭起来的——这也是为什么本教程从 [2.4 PyTorch 基础](../ch2-round1/2-2-4-pytorch-basics.md) 开始就一路用它。赛场会提供一套预装好的 Python 环境，常见的工具基本都在：

- 深度学习：`torch`、`torchvision`、`torchaudio`
- 经典机器学习与数据：`numpy`、`pandas`、`scikit-learn`、`scipy`
- 预训练模型生态：Hugging Face 的 `transformers`、`datasets`、`tokenizers`
- 图像与可视化：`Pillow`、`opencv-python`、`matplotlib`

这里有一条最关键、也最容易让人栽跟头的规则：**赛场通常是离线的，你不能在比赛中临时 `pip install` 新包，也不能从网上下载预训练权重。** 换句话说，凡是你方案里要用到的预训练模型，要么是环境里已经内置的、要么是题目附带提供的——指望现场 `from_pretrained` 去 Hugging Face 拉权重，几乎一定会失败。所以备战时你就要养成习惯：先确认目标权重是否在"允许清单"里，把方案建立在能拿到的资源上。

至于哪些是明确**不允许**的：调用外部在线 API（比如远程的大模型接口）、联网搜索、以及任何把数据传到赛场外部的行为，都属于违规。比赛要的是你**本地、独立**解决问题的能力。

## 硬件约束：你手里到底有多少算力

理解硬件，本质上是理解"你能开多大的模型"。IOAI 的实操题在带 GPU 的远程机器上完成，近几届的配置大致是单卡，显存量级在 **16 GB 上下**（具体型号每年不同，常见是数据中心级的单张 GPU）。这个数字决定了一连串现实选择：

显存 16 GB 意味着，一个十几亿参数（约 1.3B）以内的模型，用半精度（FP16/BF16）加载是可行的；再大就要靠技巧——比如冻结大部分层只微调一小块、用更小的 batch、或者干脆选更轻量的骨干网络。**所以备战时一个很重要的纪律是：始终在和赛场相近的显存预算下练手**，别在本地用一张 80 GB 的卡跑通方案，否则你对"这个模型放不放得下"会完全没有概念。

时间同样是硬约束。每道实操题有明确的时间窗（通常几个小时），而 GPU 上训练一个模型动辄几十分钟。这就逼着你**学会估算训练时长**：上场前先想清楚跑几个 epoch、要不要用预训练权重省时间，而不是上来就盲目开训，结果时间到了模型还没收敛。

## 提交系统：分数是怎么算出来的

实操题大多是这样运作的：题目给你训练数据和一个**评测脚本/评测指标**，你训练好模型后，按规定格式提交——可能是提交模型在测试集上的**预测结果文件**，也可能是提交你的代码和模型，由系统在隐藏测试集上跑分。系统拿你的输出和标准答案比对，按指标（准确率、F1、IoU 等，看题目而定）算出分数。

有几个习惯能帮你不丢冤枉分：

- **早交、多交。** 大多数赛制允许在时间窗内多次提交、取最好成绩。所以哪怕模型还没调到最优，也先交一版能跑通的"保底结果"，确认格式没问题、分数能正常显示——这一步排除了"最后一刻才发现交不上去"的灾难。
- **死磕输出格式。** 列名、文件名、ID 顺序、数值范围，任何一处和要求不符，系统都可能判为零分或解析失败。交之前一定对着题目示例文件逐项核对。
- **别等到最后一秒。** 网络和系统在结束前常常拥堵，留出几分钟提交缓冲，比什么都实在。

## 申诉流程：出了岔子怎么补救

现场难免有意外——题目描述有歧义、评测脚本疑似有 bug、机器卡死。IOAI 设有正式的**申诉（appeal / clarification）渠道**：比赛期间，你对题意的疑问可以通过官方指定的提问系统提交，由裁判统一答复；如果你认为某道题的判分或环境出了问题，可在规定时限内提出申诉，由科学委员会（Scientific Committee）裁定。

要让申诉有效，关键是**留好证据、讲清事实**：什么时间、做了什么操作、看到什么报错，越具体越好。情绪化的抱怨不会加分，清晰的复现步骤才会。还有一点要心里有数——申诉有时间窗，赛后再喊往往就来不及了，所以发现问题当下就要走流程。

## 现场要用的工具：提前练成肌肉记忆

赛场一般通过**远程开发**的方式让你连上那台 GPU 机器，主流是两条路，你应该都先在本地演练熟：

第一条是 **VS Code Remote（SSH / Remote-SSH）**：在自己电脑的 VS Code 里连上远程机器，像编辑本地文件一样写代码、开终端、跑脚本。它的好处是你熟悉的快捷键、插件、调试器都在，效率最高。

第二条是 **Jupyter Notebook / JupyterLab**：在浏览器里分单元格交互式地写代码、看中间结果。它特别适合"边训边看"——可视化一下数据、打印一下 loss 曲线、临时调个超参，反馈很快。很多 IOAI 实操题本身就以 Notebook 形式下发。

不管走哪条路，有几件事建议你现在就练顺：

```bash
# 1. 确认拿到的是哪块卡、显存多大——上场第一件事就该看它
nvidia-smi

# 2. 确认关键库的版本，别假设和你本地一样
python -c "import torch; print(torch.__version__, torch.cuda.is_available())"
```

```python
# 3. 训练前显式把模型搬上 GPU，并按显存预算控制 batch
import torch

device = "cuda" if torch.cuda.is_available() else "cpu"
model = model.to(device)              # 模型上卡
# 若加载报 "CUDA out of memory"，优先调小 batch_size、改用半精度
model = model.half()                  # 转 FP16，省一半显存
```

上面这几行不是花架子。`nvidia-smi` 让你第一时间摸清算力底牌；版本检查能在第一分钟就暴露环境差异，省得你写到一半才撞上 API 不兼容；而把模型 `.to(device)`、必要时 `.half()`，正是应对显存约束的标准动作。把这套流程练到不假思索，你上场后就能把宝贵的几个小时全花在解题上，而不是和环境搏斗。

## 一张表，把规则速查一遍

临场前扫一眼下面这张表，就能快速对号入座：

| 维度 | 现场情况 | 你该做的准备 |
| --- | --- | --- |
| 库 | PyTorch + HF + sklearn 等预装，**禁止现场装包/联网** | 方案只依赖能拿到的库和权重 |
| 硬件 | 单 GPU，显存约 16 GB，限时 | 在相近显存预算下练手，会估训练时长 |
| 提交 | 多次提交取最优，按指标自动判分 | 早交保底、死磕格式、留缓冲时间 |
| 申诉 | 官方提问/申诉渠道，有时限 | 当场留证据、讲清复现步骤 |
| 工具 | VS Code Remote / Jupyter | 两种都预先演练顺手 |
