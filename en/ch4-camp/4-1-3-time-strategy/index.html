    1.3 限时解题策略¶

    难度 ⭐⭐☆☆☆ · 前置：1.6 集成方法、1.2 调参与实验管理

    本页目标

    • 给一套可照搬的赛时时间分配模板（读题 10% / 原型 40% / 调优 40% / 复查 10%），并说清每一段该干什么
    • 学会先用最快的方式搭出一个能跑通、能提交的 baseline，把"有分"这件事尽早拿到手
    • 掌握三个性价比最高的涨分动作：early stopping、模型集成、提前备好代码模板

    为什么限时比赛拼的是"节奏"而不是"巧思"¶

    到了集训营这一关，你的知识储备其实已经够用了——CV、NLP、经典 ML 该会的都会了。可一旦坐进真正的限时赛场，很多人还是会翻车。原因往往不是不会做，而是时间没分配好：在一个想不通的细节上磕了一个钟头，回过神来发现连一次完整的提交都还没交上去。

    所以在这一关，我们要把"解题"当成一项工程来管理。你可以把一场限时赛想成一次蒙眼下山，时间就是你脚下的体力——它只会越用越少，绝不会回头。聪明的做法不是一开始就冲向你以为的最低点，而是先稳稳地往下走一段，确保自己任何时刻都站在一个"还不错"的位置上。下面这套节奏，就是帮你做到这一点的。

    一套可照搬的赛时时间分配模板¶

    假设比赛时长是 \(T\)，最稳妥的分配是把它切成四段。这个比例不是拍脑袋来的，而是无数次实战里"既不慌、又能拿高分"的经验值：

    阶段 占比 这段时间在干什么
    读题 + 探数据 10% 把题目和评分标准读透，看一眼数据长什么样
    搭原型（baseline） 40% 用最快的方法跑通"读数据 → 训练 → 提交"全流程
    调优 40% 在能提交的基础上，一点点把分数往上抬
    复查 10% 检查提交格式、留出最后一次安全提交的余量

    举个例子，如果比赛是 3 小时（180 分钟），那大约就是：读题探数 18 分钟、搭原型 72 分钟、调优 72 分钟、复查 18 分钟。你不需要拿秒表卡死，但心里得有这根弦——到了哪个时间点，就该强迫自己切换到下一阶段，哪怕上一阶段还没做到完美。

    这里藏着一个新手最容易踩的坑：把读题和探数据这 10% 省掉，上来就写代码。结果常常是做到一半才发现自己理解错了评分指标（比如以为是准确率，其实是 F1），前面的努力全打了水漂。所以读题这 10% 看着"不产出"，却是回报率最高的一段时间。

    先抢一个能提交的 baseline，别追求漂亮¶

    整套策略里最重要的一条心法是：尽早拿到一次有效提交。在很多赛制里，没交上去等于零分，哪怕你脑子里有个 90 分的方案。所以在搭原型阶段，你的唯一目标不是"做得好"，而是"跑得通、交得上"。

    具体怎么做？用你最熟、最不容易出错的工具，搭一条从头到尾能走通的最短路径。表格题就直接上一个 集成方法里学过的梯度提升树，图像题就拉一个预训练模型做迁移学习——先别想着自己从零搭网络。

    # 表格类任务的"30 行 baseline"：目标是先有一个能提交的结果
import pandas as pd
from sklearn.ensemble import HistGradientBoostingClassifier
from sklearn.model_selection import cross_val_score

train = pd.read_csv("train.csv")
test = pd.read_csv("test.csv")

# 最省事的特征：先只用数值列，类别/缺失值后面再精修
X = train.select_dtypes("number").drop(columns=["label"])
y = train["label"]

model = HistGradientBoostingClassifier()        # 自带处理缺失值，几乎不用调就能跑
print("CV 分数：", cross_val_score(model, X, y, cv=3).mean())  # 先看一眼水平

model.fit(X, y)
pred = model.predict(test.select_dtypes("number"))
pd.DataFrame({"id": test["id"], "label": pred}).to_csv("sub.csv", index=False)  # 立刻提交

    跑通这段、提交成功的那一刻，你就给自己买了一份"保险"——后面无论怎么折腾，分数都不会归零。有了这条底线，你才敢在调优阶段大胆尝试。

    调优阶段：先做性价比最高的几件事¶

    进入调优阶段，时间已经过去一半，你得学会"挑大的捡"。下面这几个动作，几乎在每一场比赛里都是稳赚不赔的。

    第一件，用 early stopping 省时间又防过拟合。 训练神经网络或提升树时，别傻乎乎地把轮数设死。盯着验证集分数，一旦它连续好几轮不再变好，就立刻停下——这既避免了模型在训练集上学过头（过拟合），又把省下的时间还给了你。

    # 以 PyTorch 风格的训练循环为例，加上 early stopping
best, patience, wait = float("inf"), 5, 0
for epoch in range(100):
    train_one_epoch(model)
    val_loss = evaluate(model)            # 在验证集上看一眼
    if val_loss < best:                   # 有进步：记下来，耐心归零
        best, wait = val_loss, 0
        torch.save(model.state_dict(), "best.pt")  # 顺手存下最好的一版
    else:                                 # 没进步：耐心 -1
        wait += 1
        if wait >= patience:              # 连续 5 轮没进步就收手
            print(f"在第 {epoch} 轮提前停止")
            break

    第二件，用集成（ensemble）再榨一点分。 这是限时赛里最经典的"免费午餐"。原理很朴素：几个各有各小毛病的模型，把它们的预测平均一下，错误往往能互相抵消，整体反而更稳。你不需要新知识，只要把手头训好的几个模型的输出取个平均：

    import numpy as np
# 把同一个任务下几个模型的预测概率求平均，再取最终类别
probs = [m.predict_proba(X_test) for m in [model_a, model_b, model_c]]
final = np.mean(probs, axis=0).argmax(axis=1)

    为什么这招在赛场上特别值？因为它几乎不需要额外的"想法"，只要你前面认真训了几个模型，临门一脚把它们一平均，常常就能再涨一两个点。集成的细节和原理，1.6 集成方法里讲得很透，赛前一定要练熟。

    第三件，每次只改一个变量。 调优时手别乱。一次只动一个东西（换个学习率、加个特征），跑完看分数再决定留不留。这套"控制变量"的纪律，和调参与实验管理里讲的完全是一回事——赛场上没有时间让你在一堆同时改动里猜是哪个起了作用。

    赛前就把模板备好，赛时只填空¶

    最后这条，是真正拉开差距的地方：比赛拼的不是临场写代码的速度，而是你赛前准备得有多充分。 顶尖选手进考场前，电脑里早就躺着一套套写好的代码模板，赛时只是把数据路径、列名一填就能跑。

    建议你赛前整理出这样一个"工具箱"：

    • 一份表格题模板：读 CSV、自动处理缺失值和类别列、交叉验证、训出提升树、生成提交文件，一条龙。
    • 一份图像题模板：加载预训练模型、改最后一层、标准的训练循环（自带 early stopping）、推理出结果。
    • 一份通用工具片段：固定随机种子、计时器、把几个模型预测做集成的几行代码。
    • 一份提交检查清单：列名对不对、行数对不对、有没有 NaN、格式和样例提交一致吗。

    把这些提前练熟、存好，到了赛场你的脑子就能腾出来专心想"这道题特殊在哪、该怎么针对性优化"，而不是耗在敲那些每次都一样的样板代码上。

    容易踩的坑¶

    • 死磕一个难点，忘了看表。 如果一个问题卡了超过你计划时间的两成还没头绪，先放下、去做能拿分的部分，回头有空再啃——别让一道题拖垮整场。
    • 从头到尾只在最后才提交一次。 万一最后那次格式错了、超时了，你就一分没有。正确做法是一有能跑的结果就先交一版，之后每次涨分了就覆盖提交。
    • 复查时间被调优吃掉。 调优是个无底洞，很容易上头。务必守住最后 10%，留给检查提交格式和做一次"安全提交"。
    • 集成时把"坏模型"也平均进去。 集成不是越多越好，一个明显很差的模型会把平均值往下拽。只平均那些各自都还不错的模型。
    • early stopping 的 patience 设得太小。 设成 1～2 容易被验证集的正常波动骗到、过早收手；一般 5 左右更稳。

    它在后面会怎么用到¶

    这套节奏不是纸上谈兵，下一节起的真题精讲就是练它的地方。无论是 2.1 CV 类真题、2.2 NLP 类真题还是 2.3 信号·优化类真题，建议你都掐着表、按"读题→原型→调优→复查"的流程走一遍。等进了 3. 团队挑战和 4. 选拔测试准备，时间和分工的压力只会更大，这套方法就更离不开了。它和误差分析与调试、调参与实验管理合在一起，正好凑成集训营的"工程三件套"。

    练习¶

    基础练习
    1. 假设比赛时长是 4 小时，按 10% / 40% / 40% / 10% 算出四个阶段各占多少分钟，并为自己写一个简单的赛时计划表。
    2. 拿一份手头的表格数据，限时 30 分钟，逼自己跑通"读数据 → 训练 → 生成提交文件"的最短路径，体会"先抢一次提交"是什么感觉。
    进阶练习
    1. 给你的训练循环加上 early stopping（patience 设为 5），对比加之前和加之后的训练耗时与验证集分数。
    2. 训练三个不同的模型（比如不同随机种子或不同算法），把它们的预测概率做平均集成，看看集成后的分数是不是比单个模型都高。

    小结¶

    一句话记住这一节：限时赛拼的是节奏和准备，不是临场的灵光一闪——先按 10/40/40/10 分好时间，尽早抢一次能提交的 baseline，再用 early stopping 和集成稳稳涨分，而这一切都建立在赛前备好的代码模板之上。

    想看真实赛场是怎么打的，可以读一读 Kaggle 高分选手的 Competition Solutions 复盘，留意他们几乎都遵循"先 baseline、再迭代、最后集成"这条主线。
